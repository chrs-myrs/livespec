# Distribution Pattern

## Status

**Updated for ambient architecture.**

Key changes:
- Plugin provides methodology (skills, metaspecs, templates) - no copying
- Projects generate CLAUDE.md (inline rules) + AGENTS.md (project context)
- No dist/ directory in target projects
- No .livespec/ directory copying
- No submodules or symlinks needed

## Requirements

- [!] LiveSpec distributes methodology through Claude Code plugin, with project-specific context generated locally
  - **Plugin Provides** (read at runtime, not copied):
    - Skills (/livespec:go, /livespec:init, /livespec:design, /livespec:audit, /livespec:learn)
    - Metaspec templates for spec creation
    - Standard conventions and guides
  - **Project Generates** (via /livespec:init):
    - CLAUDE.md with inline essential rules (spec-first + layer boundaries)
    - AGENTS.md with project-specific context
    - specs/ directory structure
    - ctxt/ sub-agent contexts (optional, for large projects)
  - No .livespec/ directory in target projects
  - Plugin version controls methodology version
  - Project context regenerates from workspace specs when needed

## Distribution Structure

### Plugin Structure (LiveSpec Repository)

```
livespec/
├── skills/           # Plugin skills (distributed via plugin)
│   ├── go/          # Intelligent router
│   ├── init/        # Bootstrap
│   ├── design/      # Spec creation
│   ├── audit/       # Health checks (was: evolve)
│   └── learn/       # Session capture
├── references/       # Templates and guides (read at runtime)
│   ├── templates/   # Metaspec templates
│   ├── guides/      # Usage guides
│   └── standard/    # MSL metaspecs
└── specs/           # LiveSpec's own specs (dogfooding)
```

### Project Structure (Target Projects)

```
your-project/
├── PURPOSE.md        # Generated by /livespec:init
├── CLAUDE.md         # Generated with inline rules
├── AGENTS.md         # Generated from workspace specs
├── specs/
│   ├── workspace/   # How you work
│   ├── foundation/  # Why (outcomes, constraints)
│   ├── strategy/    # How (architecture)
│   ├── features/    # What (behaviors)
│   └── interfaces/  # Contracts
└── ctxt/            # Optional sub-agent contexts
```

**Note:** No .livespec/ directory. Plugin provides methodology at runtime.

## Methodology Immutability

**Core principle:** Plugin provides immutable methodology reference

**Plugin is read-only:**
- Installed via `claude plugin install livespec`
- Skills invoked as /livespec:* commands
- Templates read when creating specs
- Updates via plugin update

**Project customization via specs/workspace/:**
- constitution.spec.md - Your development principles
- patterns.spec.md - Your coding conventions
- workflows.spec.md - Your development processes
- taxonomy.spec.md - Your domain classification

**Methodology doesn't fit?**
- Fork plugin repository, install from local path
- Normal case: Customize via specs/workspace/ (99% of projects)

**What this prevents:**
- ❌ Modification drift (plugin is canonical)
- ❌ Stale methodology (plugin updates automatically)
- ❌ Confusion about source (plugin is single source)

**What's preserved:**
- ✅ specs/workspace/ - Your project-specific process
- ✅ CLAUDE.md - Your inline rules (regenerable)
- ✅ AGENTS.md - Your generated context (regenerable)

### Context Generation Pattern

- **CLAUDE.md** - Generated by /livespec:init with inline essential rules
  - Spec-first protocol (check spec exists before implementation)
  - Layer boundary rules (foundation → strategy → features)
  - Project-specific configuration
- **AGENTS.md** - Generated from workspace specs (20-30KB, routing)
  - Customized to project PURPOSE, domain, constraints
  - Root routing to specialized sub-agents
  - Total context <60KB typical
- **ctxt/** - Optional sub-agent contexts (for large projects)
  - ctxt/phases/ - Phase specialists (define, design, evolve)
  - ctxt/domains/ - Domain-specific patterns
  - ctxt/utils/ - Utility specialists (session, msl-audit)
- **Generation**: Via /livespec:init (initial) or /livespec:audit (regenerate)
- **Timing**: Bootstrap creates initial, workspace spec changes trigger regeneration

**Generated content characteristics:**
- No frontmatter (clean agent context)
- Inline edit warning at top
- Customized to project PURPOSE, domain, constraints
- References plugin for detailed methodology

## Deprecated Methods (For Reference Only)

The following methods are superseded by plugin installation but documented for migration:

### Sparse Submodule (DEPRECATED)
- Created .livespec-repo/ pointing to livespec repository
- Symlinked .livespec → .livespec-repo/dist
- **Migration**: Remove submodule, remove symlink, install plugin

### Directory Copy (DEPRECATED)
- Copied dist/ contents to .livespec/
- **Migration**: Remove .livespec/, install plugin

### Symlink Architecture (DEPRECATED)
- Used prompts/ symlinks to .livespec/prompts/
- Used prompts/generated/ for project-specific content
- **Migration**: Remove symlinks, /livespec:init generates context

## Template Architecture Pattern

**Purpose:** Reusable markdown content for consistent spec creation

**When to Create Template:**
- Content reused across multiple generated files
- Key process/checklist needing consistent application
- Standard spec formats (metaspecs)

**Template Structure:**
- **Location**: `references/templates/{domain}/` in plugin (e.g., `metaspecs/`, `agents/`)
- **Format**: Complete standalone markdown (not fragment)
- **Naming**: Descriptive (e.g., `behavior.spec.md`, `contract.spec.md`)

**Template Flow:**
1. **Templates in plugin** provide reusable content (`references/templates/*.md`)
2. **Skills read** templates at runtime when creating specs
3. **Generated specs** follow template structure
4. **Plugin updates** automatically provide latest templates

**Example:** Metaspec templates
- `references/templates/metaspecs/behavior.spec.md` - Template for behavior specs
- Read by: `/livespec:design feature <name>`
- Result: New spec follows standard structure

**Benefits:**
- Templates always current (plugin provides latest)
- No copying or syncing needed
- Consistent across all projects using same plugin version

## Reference Library Pattern

**Purpose**: Enable agents to navigate from AGENTS.md summary (80%) to deep detail (20%)

**Path convention**: Plugin provides detailed guides at runtime
- Skills can read detailed guides when needed
- AGENTS.md provides routing summary, plugin provides depth
- Consistent across all environments using same plugin

**Structure**:
- **Categories**: Conventions (how to structure), Metaspecs (spec types), Guides (how to apply)
- **Entry format**:
  - **Fetch when**: Action-oriented trigger conditions
  - **Provides**: What detail this reference contains
  - **Via**: Skill or direct plugin reference

**Example**:
```markdown
**MSL Minimalism** - `/livespec:design` provides guidance
- **Fetch when**: Creating new specs, reviewing spec quality
- **Provides**: Four-question test, decision framework
- **Via**: Skill loads guide from plugin when invoked
```

**Location**: MIDDLE or END section of AGENTS.md (detailed navigation, not critical rules)

**Integration**: Cross-reference from AGENTS.md to skills
```markdown
## MSL Minimalism
[Brief summary]

**Full framework**: Use `/livespec:design` for guided spec creation
```

## Validation

### Plugin Distribution
- [ ] Plugin installs without errors
- [ ] Skills available after install (/livespec:go, /livespec:init, etc.)
- [ ] Templates readable from plugin path
- [ ] No .livespec/ directory created in target projects

### Project Generation
- [ ] /livespec:init generates CLAUDE.md with inline rules
- [ ] /livespec:init generates AGENTS.md with project context
- [ ] /livespec:init creates specs/ directory structure
- [ ] /livespec:audit regenerates context when workspace specs change
- [ ] Generated files have no frontmatter (clean context)

### Content Separation
- [ ] Plugin provides: skills, templates, guides (not copied)
- [ ] Project generates: CLAUDE.md, AGENTS.md, ctxt/ (committed or gitignored per preference)
- [ ] specs/workspace/ contains project customization
- [ ] No drift possible (plugin is single source of methodology)
